Detailed project guide

Worldcup Manager 2026

Worldcup Manager 2026 is a learning project for managing a fixed football competition. The app is intentionally small enough to study, but still shows a realistic full-stack flow: authentication, role checks, API services, tournament progression, score handling, and a multilingual UI.

Express + TypeScript backend Prisma + PostgreSQL Next.js frontend JWT auth + roles i18n: nl / en / fr
What changed in the cleanup The codebase now uses one canonical competition route, removes dead dashboard exports, and documents the tournament flow more clearly. Main overview route: /api/competition

1. Technology Stack

Backend

  • Node.js with Express and TypeScript.
  • Prisma ORM for database access and typed queries.
  • PostgreSQL as the persistent data store.
  • JWT-based authentication with role-aware middleware.
  • Swagger UI for generated API documentation.

Frontend

  • Next.js Pages Router with React and TypeScript.
  • SWR for client-side data fetching and cache revalidation.
  • Custom i18n context with persisted language preference.
  • Session state stored in localStorage and restored on load.

3. Domain Model

The application models one fixed competition. The data is intentionally constrained so the logic stays readable for students.

User

Represents a login account. Roles decide what the user can see or change.

Team

A national team with country metadata, flag, and competition statistics.

Player

Belongs to one team and can be marked available or unavailable.

Match

Stores the fixture, teams, referee, score, date, round, and status.

Goal

Links a scorer to a match so the score can be recalculated from the stored events.

Round

Derived from the fixed competition setup; rounds are not free-form user content.

The important match states are NOT_STARTED, IN_PROGRESS, FINISHED, and COMPLETED.

4. Backend Architecture

Request flow

  1. app.ts configures Express, Helmet, CORS, JSON parsing, Swagger, and routers.
  2. Controllers translate HTTP requests into service calls.
  3. Services implement business rules, validation, and database access.
  4. Models normalise Prisma records into API-friendly response shapes.
  5. Utilities handle cross-cutting concerns such as JWT verification and error mapping.

Canonical backend route map

  • /status for health checks.
  • /api/auth for login, register, and the current user.
  • /api/competition for the overview and round simulation.
  • /api/players, /api/matches, and /api/users for domain actions.

Why the layered structure matters

The backend is split so the HTTP layer stays thin. That makes the business logic easier to test and easier to explain: controllers answer what endpoint was called, services answer what the system is allowed to do, and models answer how records should look.

Important backend files

File Role
back-end/app.tsStarts the server and mounts middleware and routers.
back-end/controller/*.tsExpose the REST API.
back-end/service/*.tsContain the real competition logic.
back-end/model/*.tsValidate and transform domain objects.

Teacher explanation tip

If you are asked why the backend has models and services instead of putting everything in the controllers, say that the project is deliberately structured to separate transport concerns from domain rules. That keeps the app testable and stops the frontend from becoming the place where rules are enforced.

5. Security and Access Control

Authentication

  • Users log in with a username and password.
  • The server returns a JWT token on success.
  • The frontend stores the token and user in localStorage.
  • On refresh, the session provider restores the stored values.

Authorization

  • authenticateToken checks the bearer token header.
  • requireRoles blocks routes that do not match the user's role.
  • getAuthenticatedUser centralises the repeated request-user extraction.
  • Admins and referees are protected on match editing routes.

Extra security middleware such as Helmet and CORS is configured in back-end/app.ts. The CORS origin is environment-based, so the frontend can be locked down per setup.

6. Competition Flow

How the bracket progresses

  1. The overview loads all rounds, teams, matches, goals, standings, and top scorers.
  2. Admins or referees update match status and scores on their allowed matches.
  3. When a round is fully complete, createNextRoundMatchesIfReady pushes the winners forward.
  4. If the next round still has no teams assigned, it is filled from the winners list inside a transaction.
  5. Completed rounds are marked as such so the UI can show clean stage progression.

Why simulation exists

Simulation is a learning helper. It lets a teacher or student show the full lifecycle without manually entering every score. The service simulates non-draw knockout results and assigns goal scorers from the players available for that team.

Rules that are intentionally enforced

  • Knockout matches cannot end in a draw.
  • Goals can only be entered when a match is in progress.
  • Referees can only act on their assigned matches.
  • Previous rounds must be completed before later rounds can be edited or simulated.

7. Frontend Architecture

Core pieces

  • pages/_app.tsx wraps the app with i18n, session, and layout providers.
  • lib/useSession.tsx restores auth state from localStorage and exposes login/logout helpers.
  • lib/i18n.tsx stores the current language and translation map.
  • services/*.ts keep fetch logic out of the page components.
  • components/DashboardPanels.tsx now only exports the score leaderboard panel that the stats page needs.

Why the UI is easy to follow

Each page mostly decides what data to show and when to block access. The actual HTTP calls are in service files, so the page components stay readable and the data flow is easy to present.

8. Internationalisation

Supported locales

  • nl for Dutch.
  • en for English.
  • fr for French.

How the language state works

  • The selected locale is stored in localStorage.
  • useI18n provides t(), the locale, and a setter for the language.
  • The language switcher updates the active locale without a full page reload.

This is a good section to mention during presentation because it shows state that is not server-driven: the chosen language survives refresh, but the competition data itself still comes from the backend.

9. Local Setup

Backend

  1. Start PostgreSQL with Docker.
  2. Install backend dependencies.
  3. Generate the Prisma client.
  4. Seed the database.
  5. Start the backend server.

Frontend

  1. Set NEXT_PUBLIC_API_BASE_URL if the backend runs on a custom port.
  2. Install frontend dependencies.
  3. Run the Next.js dev server.

The repo memory notes already capture one important setup detail: if port 3000 is occupied, move the backend to another port and keep the frontend API base URL aligned.

10. How to Explain the Code

Start with the problem

The app manages a competition with users, teams, matches, scores, and roles.

Then explain the split

The backend owns the rules, the frontend owns the presentation, and the database stores the state.

Finish with the flow

Login, load overview, inspect matches, edit a match, complete a round, and advance winners.

If a teacher asks why the code is structured this way, the short answer is that the project is intentionally layered so the rules are visible, testable, and easy to walk through on screen.

12. Glossary

Source of truth: the backend and database are the authoritative state for matches, goals, and users.

Role guard: backend check that allows or blocks a request based on the authenticated user's role.

Revalidation: SWR refetches data after a mutation so the UI stays in sync.

Bracket progression: automatic creation and population of the next round after a round finishes.

i18n: internationalisation, the mechanism used here for Dutch, English, and French translations.

Lean refactor: the cleanup that removed duplicate routes and dead dashboard exports.
