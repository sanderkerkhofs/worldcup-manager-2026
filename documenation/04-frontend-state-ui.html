04 - Frontend, State, and UI Responsibilities

1. Session and Auth State

useSession manages token, user object, auth state, and login/logout actions. Session is persisted and rehydrated from browser storage.

2. Data Fetching Strategy

SWR is used for caching, revalidation, and predictable async UI states (isLoading, error, mutate).

3. Role-Based Pages

  • /admin: user management and round access
  • /coach: player availability updates
  • /referee: assigned match workflow
  • /matches/[matchId]: detailed match editing
  • /rounds/[roundId]: stage controls and list

/rounds redirects to / because bracket overview is centralized on dashboard.

4. UI Design Pattern

  • heroCard for top-level context
  • panelCard for operations
  • tableWrap for responsive tables

5. Status Vocabulary

  • NOT_STARTED
  • IN_PROGRESS
  • COMPLETED

Frontend and backend are aligned on this status model.

6. Service Mapping Examples

  • getOverview() -> /api/competition
  • updateMatchStatus() -> /api/matches/:matchId/status
  • updatePlayerStatus() -> /api/players/:playerId/status
  • simulateRound() -> /api/competition/rounds/:roundId/simulate

Round endpoints remain, but roundId is a stage identifier instead of a persisted round entity id.

7. Debugging Path

  1. Check page SWR key and state.
  2. Check called frontend service method.
  3. Check backend controller/service pair.
  4. Validate domain rule expectations.
