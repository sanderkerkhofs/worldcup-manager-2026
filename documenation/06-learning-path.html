06 - Learning Path (How to Study This Project)

Phase 1 - Big Picture

Read 01 and 02. Target: explain architecture + data model in 2-3 minutes.

Phase 2 - Business Rules

Read 03 and 05. Target: justify the key knockout validations.

Phase 3 - UI Behavior

Read 04. Target: trace one click from UI to API to DB update.

Phase 4 - Demo Narratives

  1. How stage status moves NOT_STARTED -> IN_PROGRESS -> COMPLETED
  2. How referee updates are safely constrained
  3. How coach availability drives match operations
  4. How role-based access is enforced

Phase 5 - Self-Check Questions

  1. Why both controller and service validation?
  2. Why store goals as events instead of only final score?
  3. What prevents invalid knockout outcomes?
  4. Why SWR over direct fetch in every component?
  5. What breaks if IN_PROGRESS is skipped before result entry?

Phase 6 - Reflection Ideas

  • Add audit logging for critical role actions
  • Add optimistic UI updates in match editor
  • Add pagination for larger datasets
  • Add integration tests for role workflows
  • Define production deployment profile

Quick Revision Sheet

  • Architecture: layered backend + service-centric rules
  • Data model: User, Team, Player, Match (with stage metadata), Goal
  • Statuses: NOT_STARTED -> IN_PROGRESS -> COMPLETED
  • Security: JWT + role middleware + contextual checks
