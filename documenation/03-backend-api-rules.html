03 - Backend API and Business Rules

1. API Areas

Auth

  • POST /api/auth/register
  • POST /api/auth/login
  • GET /api/auth/me

Competition

  • GET /api/competition/overview
  • GET /api/competition/rounds
  • POST /api/competition/rounds/:roundId/initiate (ADMIN)
  • POST /api/competition/rounds/:roundId/simulate (ADMIN)

Teams / Players / Matches / Users

Domain management endpoints are available under /api/teams, /api/players, /api/matches, and /api/users.

2. Key Business Rules

Round Initiation

  • Round must exist.
  • Previous round must be completed (except first round).
  • Matches must have both teams known.
  • Status moves from NOT_STARTED to IN_PROGRESS.

Round Simulation

  • Cannot run if NOT_STARTED matches remain.
  • Teams must be assigned.
  • Simulation creates non-draw outcomes with goals.

Match Update

  • Only ADMIN or assigned REFEREE can edit status/result/goals.
  • IN_PROGRESS/COMPLETED require known teams.
  • COMPLETED requires valid non-draw knockout score.

Goal Rules

  • Player must exist and be available.
  • Goal team must match player team.
  • Goal team must be one of match teams.

3. Security Model

  • JWT validation on protected routes.
  • Role middleware enforces permissions.
  • Service checks enforce context (assigned referee, team ownership).

4. Error Strategy

Typed errors keep API responses consistent: ValidationError, NotFoundError, ForbiddenError, UnauthorizedError.

5. Case-Insensitive Login

Username lookup in login is case-insensitive, improving usability while preserving account uniqueness.

6. API Contract

Use Swagger at /api-docs for endpoint contracts and payload schemas.
