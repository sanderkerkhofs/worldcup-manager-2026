REST API Endpoints

Every route the back-end exposes, with explanations of what they do.

What is a REST API?

REST (Representational State Transfer) is a convention for building APIs using HTTP. The key ideas are:

  • Resources — things you work with: /api/matches, /api/players, /api/users
  • HTTP Methods — what you do with a resource:
MethodMeaningExample
GETRead / retrieve dataGET /api/matches → list all matches
POSTCreate a new resourcePOST /api/auth/register → create new user
PUTReplace a resource completelyPUT /api/matches/:id/result → set final score
PATCHPartially update a resourcePATCH /api/matches/:id/status → change only the status
DELETERemove a resourceDELETE /api/users/:id → delete a user

URL parameters like :matchId are placeholders — the actual ID goes in the URL, e.g. /api/matches/clh3x8abc.

Legend

🔓 Public — no login required   🔑 Auth — must be logged in   👮 REFEREE — must have REFEREE role   ⚙️ ADMIN — must have ADMIN role

Auth endpoints — /api/auth

POST /api/auth/register 🔓 Public

Registers a new user account with the USER role.

Request body:

{ "username": "john_doe", "password": "mypassword123" }

Response (201):

{ "user": { "id": "...", "username": "john_doe", "role": "USER" }, "token": "eyJ..." }

Errors: 400 if username is taken or input is invalid.

POST /api/auth/login 🔓 Public

Authenticates a user and returns a JWT token.

Request body:

{ "username": "john_doe", "password": "mypassword123" }

Response (200):

{ "user": { "id": "...", "username": "john_doe", "role": "USER", "teamId": null }, "token": "eyJ..." }

Errors: 401 if credentials are wrong.

GET /api/auth/me 🔑 Auth

Returns the currently logged-in user's information. Used by the front-end to verify a stored token is still valid.

Response (200): same user object as login.

Competition endpoints — /api/competition

GET /api/competition 🔓 Public

Returns a full competition overview: metadata, all rounds, all matches, all teams, standings, and top scorers. This is the main data endpoint used by the home page.

Response:

{
  "competition": { "name": "World Cup 2026", "year": 2026, "hostCountry": "USA", ... },
  "rounds": [{ "id": "1", "name": "8th Final", "orderNumber": 1, ... }, ...],
  "matches": [{ "id": "...", "status": "PLANNED", "homeTeamId": "...", ... }, ...],
  "teams": [{ "id": "...", "name": "Argentina", "countryFlag": "🇦🇷", ... }, ...],
  "standings": [{ "teamId": "...", "teamName": "Argentina", "points": 9, ... }, ...],
  "topScorers": [{ "playerId": "...", "playerName": "Messi, Lionel", "goals": 3, ... }, ...]
}
GET /api/competition/rounds 🔓 Public

Returns just the list of rounds (Round of 16, Quarterfinal, Semifinal, Final).

GET /api/competition/rounds/:roundId/matches 🔑 Auth

Returns all matches belonging to a specific round.

POST /api/competition/rounds/:roundId/simulate ⚙️ ADMIN

Simulates a round: automatically generates random scores for all unfinished matches, marks them FINISHED/COMPLETED, and advances the bracket. Used for testing/demo purposes.

POST /api/competition/reset-matches ⚙️ ADMIN

Resets all matches to their initial PLANNED state, clears all goals and scores, and resets teams in rounds 2-4 to TBD. Useful for restarting the tournament from scratch.

Match endpoints — /api/matches

GET /api/matches 🔓 Public

Returns all matches ordered by date. Does not include goal details.

GET /api/matches/top-scorers 🔓 Public

Returns a ranked list of players by goals scored across the entire tournament.

Response:

[
  { "playerId": "...", "playerName": "Messi, Lionel", "teamId": "...", "teamName": "Argentina", "goals": 4 },
  { "playerId": "...", "playerName": "Mbappé, Kylian", ..., "goals": 3 }
]
GET /api/matches/:matchId 🔓 Public

Returns a single match with full details including the goal-by-goal timeline (player name, team, timestamp).

PATCH /api/matches/:matchId/status 👮 ADMIN or REFEREE

Updates the status of a match. Referees can only change their own assigned match and only move status forward.

Request body: { "status": "IN_PROGRESS" }

Valid status flow: PLANNED → NOT_STARTED → IN_PROGRESS → FINISHED → COMPLETED

PUT /api/matches/:matchId/result 👮 ADMIN or REFEREE

Sets the final score of a match. Automatically triggers bracket advancement if the round is completed.

Request body: { "homeScore": 2, "awayScore": 1, "status": "FINISHED" }

POST /api/matches/:matchId/goals 👮 ADMIN or REFEREE

Records a goal in a match. The score is automatically recalculated from stored goals.

Request body:

{ "playerId": "...", "teamId": "..." }

Validation: player must be AVAILABLE and belong to one of the match teams.

PATCH /api/matches/:matchId/goals/:goalId 👮 ADMIN or REFEREE

Updates an existing goal record (e.g., corrects the scoring player). Score is recalculated.

Player endpoints — /api/players

GET /api/players?teamId=:teamId 🔑 Auth

Returns all players for a given team. The teamId is a query parameter (in the URL after ?).

PATCH /api/players/:playerId/status ⚙️ ADMIN

Updates a player's availability status (AVAILABLE / UNAVAILABLE).

Request body: { "status": "UNAVAILABLE" }

User management endpoints — /api/users

GET /api/users ⚙️ ADMIN

Lists all registered users with their roles. Admin only.

DELETE /api/users/:userId ⚙️ ADMIN

Deletes a user account. Admin only.

PATCH /api/users/:userId/role ⚙️ ADMIN

Changes a user's role (e.g., promotes a USER to REFEREE).

PATCH /api/users/:userId/assign-match ⚙️ ADMIN

Assigns a referee to a specific match.

Health check

GET /status 🔓 Public

Returns a simple "running" message. Used by Docker healthchecks and monitoring tools.

{ "message": "Back-end is running..." }

API Documentation (Swagger)

The back-end automatically generates an interactive API documentation page using Swagger UI. Visit http://localhost:3001/api-docs when the back-end is running.

From the Swagger UI you can:

  • Browse all available endpoints
  • See expected request/response formats
  • Try out API calls directly from the browser (with authentication)
