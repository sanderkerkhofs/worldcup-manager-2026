Chapter 07

Docker & Deployment

How the application is packaged and run as containers.

Why Docker?

Imagine sharing this project with a classmate. Without Docker, they would need to:

  1. Install Node.js (correct version)
  2. Install PostgreSQL (correct version)
  3. Create a database and user
  4. Set up environment variables
  5. Install npm packages in both front-end and back-end

With Docker: docker compose up --build and everything starts.

Docker achieves this by packaging the app into a container — a self-contained environment that includes the app code, its dependencies, and the runtime (Node.js). The container runs identically everywhere.

Key Docker concepts

Image

A read-only template (like a blueprint). An image is built from a Dockerfile. You can share images via Docker Hub.

Container

A running instance of an image. Just like a class and an object — the image is the class, the container is the instance.

Volume

A way to persist data outside the container. Used for the PostgreSQL database so data survives container restarts.

Network

Containers in the same Compose file can communicate by service name. The backend reaches the database at the hostname database.

The back-end Dockerfile

File: back-end/Dockerfile

# Use the official Node.js 18 image as the base
FROM node:18-alpine

# Set the working directory inside the container
WORKDIR /app

# Copy package files first (for better layer caching)
COPY package*.json ./
COPY prisma ./prisma/

# Install all dependencies
RUN npm install

# Generate Prisma client from the schema
RUN npx prisma generate

# Copy the rest of the application code
COPY . .

# Expose the port the app listens on
EXPOSE 3000

# Start the application
CMD ["npx", "ts-node", "app.ts"]

Why copy package.json first?

Docker builds images in layers. Each instruction is a layer. If the code changes but package.json hasn't changed, Docker can skip the npm install layer and reuse the cached one — making subsequent builds much faster.

The front-end Dockerfile

File: front-end/Dockerfile

FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

# Build the Next.js production app
RUN npm run build

EXPOSE 8080

# Start Next.js in production mode
CMD ["npm", "start"]

docker-compose-fullstack.yml explained

services:

  # ─── PostgreSQL database ────────────────────────────────────
  database:
    image: postgres:16-alpine          # use official Postgres 16 image
    container_name: tournament-database
    restart: unless-stopped            # auto-restart if it crashes
    environment:
      POSTGRES_DB: tournament_manager  # database name to create
      POSTGRES_USER: tournament        # DB user
      POSTGRES_PASSWORD: tournament    # DB password
    ports:
      - "5432:5432"                    # host:container port mapping
    volumes:
      - postgres_data:/var/lib/postgresql/data  # persist data on disk
    healthcheck:
      # only marks as "healthy" when pg_isready succeeds
      test: ["CMD-SHELL", "pg_isready -U tournament -d tournament_manager"]
      interval: 5s
      retries: 10

  # ─── Express back-end ───────────────────────────────────────
  backend:
    build:
      context: ./back-end              # build using back-end/Dockerfile
    container_name: tournament-backend
    restart: unless-stopped
    depends_on:
      database:
        condition: service_healthy     # wait until DB is healthy before starting
    environment:
      APP_PORT: 3000
      DATABASE_URL: postgresql://tournament:tournament@database:5432/tournament_manager
      # Note: "database" above is the service name — resolved by Docker's DNS
      JWT_SECRET: change-me-in-production
      JWT_EXPIRES_IN: 7d
      CORS_ORIGIN: http://localhost:8080
    ports:
      - "3001:3000"                    # expose as 3001 on the host machine

  # ─── Next.js front-end ──────────────────────────────────────
  frontend:
    build:
      context: ./front-end
    container_name: tournament-frontend
    restart: unless-stopped
    depends_on:
      - backend                        # wait for backend to start first
    environment:
      NEXT_PUBLIC_API_URL: http://localhost:3001  # browser uses this URL
    ports:
      - "8080:8080"

volumes:
  postgres_data:                       # named volume for DB persistence

Port mapping explained

ServiceContainer portHost portAccess from browser
database54325432localhost:5432 (DB tools)
backend30003001localhost:3001
frontend80808080localhost:8080

"3001:3000" means: forward port 3001 on your machine to port 3000 inside the container. The app inside still listens on 3000 — only the outside mapping is different.

The two docker-compose files

docker-compose-fullstack.yml

Starts all three services: database + backend + frontend. Use this to run the complete application.

docker compose -f docker-compose-fullstack.yml up --build

docker-compose-pgsql.yml

Starts only the database. Use this during development when you run the back-end and front-end locally with npm run start.

docker compose -f docker-compose-pgsql.yml up -d

Common Docker commands

CommandWhat it does
docker compose up --buildBuild images and start all containers
docker compose up -dStart in background (detached)
docker compose downStop and remove containers
docker compose down -vAlso delete volumes (wipes the database!)
docker compose logs -f backendFollow the backend container logs
docker compose psShow running containers and their status
docker compose exec backend shOpen a shell inside the backend container

Development without Docker

You can also run the project without Docker (useful for faster development with hot reload):

  1. Start only the database: docker compose -f docker-compose-pgsql.yml up -d
  2. In back-end/: create a .env file with the database URL, then run npm run db:seed and npm start
  3. In front-end/: create a .env.local with NEXT_PUBLIC_API_URL=http://localhost:3000, then run npm run dev

How Docker networking works here

When Docker Compose starts, it creates a private network for all services. Containers communicate using each other's service name as the hostname:

# In the backend environment variables:
DATABASE_URL: postgresql://tournament:tournament@database:5432/...
#                                                 ^^^^^^^^
#                          "database" = the service name in docker-compose.yml
#                          Docker resolves this to the container's IP address

The browser (outside Docker) accesses the services through the port mappings (localhost:3001, localhost:8080). But inside the Docker network, services talk to each other directly.
