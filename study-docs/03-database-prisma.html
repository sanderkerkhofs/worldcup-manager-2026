Database & Prisma

How data is stored, structured, and accessed.

What is a relational database?

A relational database stores data in tables — like spreadsheets with columns (fields) and rows (records). Tables can reference each other through foreign keys.

Example: a Goal row has a matchId column pointing to a row in the Match table. This is called a relation. You can query "give me all goals for match X" by joining the tables on that key.

The Prisma schema file

File: back-end/repository/prisma/schema.prisma

This one file defines your entire database structure. Prisma reads it and:

  1. Generates the SQL to create the database tables
  2. Generates TypeScript types so you have full type-safety when querying

The User model

model User {
  id           String    @id @default(cuid())   // primary key, auto-generated ID
  username     String    @unique                 // must be unique across all users
  passwordHash String                            // bcrypt hash, never plain text!
  role         UserRole  @default(GUEST)         // enum: ADMIN | REFEREE | USER | GUEST
  teamId       String?                           // optional FK (? = nullable)
  createdAt    DateTime  @default(now())
  updatedAt    DateTime  @updatedAt              // auto-updated on every save

  // Relations (Prisma builds the join queries for you)
  team            Team?   @relation(fields: [teamId], references: [id], onDelete: SetNull)
  assignedMatches Match[] @relation("AssignedReferee")
}

Key Prisma concepts

SyntaxMeaning
@idMarks this field as the primary key
@default(cuid())Auto-generates a unique string ID like clh3x...8
@uniqueDatabase enforces uniqueness (no duplicate usernames)
String?The ? means the field is optional (can be NULL)
@default(now())Sets createdAt to the current timestamp automatically
@updatedAtAutomatically updates the timestamp whenever the row changes
@relation(...)Defines a foreign key relationship between two tables
@@index([field])Creates a database index for faster queries on that field
@@unique([a, b])Combination of two fields must be unique (e.g., team+shirtNumber)

The Match model

model Match {
  id               String      @id @default(cuid())
  roundOrderNumber Int                             // 1 = Round of 16, 2 = QF, etc.
  roundName        String                          // "8th Final", "Quarterfinal", ...
  homeTeamId       String?                         // null = TBD (winner fills it in)
  awayTeamId       String?
  refereeId        String?                         // assigned referee user
  homeScore        Int?                            // null until match is played
  awayScore        Int?
  matchDate        DateTime
  status           MatchStatus @default(PLANNED)
  createdAt        DateTime    @default(now())
  updatedAt        DateTime    @updatedAt

  homeTeam  Team? @relation("HomeTeam", fields: [homeTeamId], references: [id], onDelete: Cascade)
  awayTeam  Team? @relation("AwayTeam", fields: [awayTeamId], references: [id], onDelete: Cascade)
  referee   User? @relation("AssignedReferee", fields: [refereeId], references: [id], onDelete: SetNull)
  goals     Goal[]

  @@index([roundOrderNumber])
  @@index([refereeId])
}

Why are homeTeamId and awayTeamId nullable? Because in a knockout tournament, you don't know the next round's teams until the current round finishes. A match is created with null teams and filled in once the winners are determined.

Database Enums

Enums are a special type that only allows specific values. They are enforced at the database level.

enum UserRole {
  ADMIN    // can do anything in the system
  COACH    // coach of a national team
  REFEREE  // assigned to specific matches
  USER     // regular fan/viewer
  GUEST    // unauthenticated public viewer
}

enum MatchStatus {
  PLANNED      // match is scheduled but not ready to start
  NOT_STARTED  // teams are confirmed, waiting for kickoff
  IN_PROGRESS  // match is currently being played
  FINISHED     // final score is entered, winner determined
  COMPLETED    // round advancement processed, bracket updated
}

enum PlayerStatus {
  AVAILABLE    // player can be selected as goal scorer
  UNAVAILABLE  // player is injured, suspended, etc.
}

Working with Prisma

CRUD operations

// CREATE — insert a new row
const user = await prisma.user.create({
  data: {
    username: 'john_doe',
    passwordHash: '$2b$10$...',
    role: 'USER',
  }
});
// user.id, user.createdAt etc. are filled in automatically

// READ — find by primary key
const match = await prisma.match.findUnique({
  where: { id: 'clh3x8...' }
});

// READ — find many with filters
const finishedMatches = await prisma.match.findMany({
  where: { status: 'FINISHED' },
  orderBy: { matchDate: 'desc' },
  take: 10,  // limit to 10 results
});

// UPDATE — change specific fields
const updated = await prisma.match.update({
  where: { id: matchId },
  data: { status: 'IN_PROGRESS', homeScore: 0, awayScore: 0 },
});

// DELETE
await prisma.user.delete({ where: { id: userId } });

// UPDATE MANY
await prisma.match.updateMany({
  where: { roundOrderNumber: 1, status: 'FINISHED' },
  data: { status: 'COMPLETED' },
});

Relations & includes (JOINs)

// Include related data (like a SQL JOIN)
const match = await prisma.match.findUnique({
  where: { id: matchId },
  include: {
    goals: {
      include: {
        player: { select: { id: true, firstName: true, lastName: true } },
        team:   { select: { id: true, name: true, countryFlag: true } },
      },
      orderBy: { createdAt: 'asc' },
    },
  },
});
// match.goals is now an array of goals with player and team data embedded

Transactions

A transaction groups multiple operations so they either all succeed or all fail together. This prevents the database from ending up in a half-updated, inconsistent state.

// When advancing the bracket: update all next-round matches together
await prisma.$transaction(async (tx) => {
  for (let i = 0; i < nextRoundMatches.length; i++) {
    // Delete any old goals in that match slot
    await tx.goal.deleteMany({ where: { matchId: nextRoundMatches[i].id } });

    // Assign the winners from the previous round
    await tx.match.update({
      where: { id: nextRoundMatches[i].id },
      data: { homeTeamId: winners[i*2], awayTeamId: winners[i*2+1] },
    });
  }
  // If ANY of the above throws, ALL changes are rolled back
});

Database seeding

File: back-end/util/seed.ts

Seeding means populating the database with initial data for development/testing. The seed script creates:

  • 16 national teams with their squads of players
  • The full knockout bracket (Round of 16 → QF → SF → Final)
  • User accounts (admin, referees, regular users)

Run it with: npm run db:seed (this also resets the database first!)

How rounds work in this tournament

This is a knockout tournament with 4 rounds:

Round orderNameMatchesTeams
1Round of 16 (8th Final)8 matches16 teams → 8 winners
2Quarterfinal4 matches8 teams → 4 winners
3Semifinal2 matches4 teams → 2 winners
4Final1 match2 teams → 1 champion

When all matches in round N are COMPLETED, the roundProgressionService automatically:

  1. Determines the winner of each match (higher score wins — draws are not allowed)
  2. Assigns the winners as home/away teams for the next round's matches
  3. Resets those next-round matches so they are ready to be played
