Chapter 04

Authentication & Security

How users log in, how sessions work, and how the app stays secure.

Authentication vs Authorisation

🔑 Authentication

"Who are you?"

Verifying that the user is who they claim to be. In this project: checking a username + password and issuing a token.

🛡️ Authorisation

"Are you allowed to do this?"

Checking if the authenticated user has the role required for the action they want to perform.

Password security: bcrypt

Passwords are never stored in plain text. Instead, they are hashed using bcrypt before being saved to the database.

What is hashing?

A hash function takes input (your password) and produces a fixed-length output (the hash). It is a one-way process — you cannot reverse a hash back to the original password.

// back-end/util/password.ts

import bcrypt from 'bcrypt';

const SALT_ROUNDS = 12; // higher = slower but more secure

// When user registers: hash the password before storing
export async function hashPassword(plainText: string): Promise {
  return bcrypt.hash(plainText, SALT_ROUNDS);
}
// "password123" → "$2b$12$5K9BrwJl5YYR5..." (irreversible)

// When user logs in: compare the entered password with the stored hash
export async function comparePassword(plainText: string, hash: string): Promise {
  return bcrypt.compare(plainText, hash); // returns true or false
}

Why not just use a regular hash like SHA256?

SHA256 is very fast — an attacker with a list of common passwords can hash all of them in seconds and compare. bcrypt is intentionally slow (controlled by the "salt rounds"). This makes brute-force attacks impractical. The salt also means the same password always produces a different hash, defeating rainbow table attacks.

JWT — JSON Web Tokens

Once a user logs in, the server issues a JWT. The client stores this token and sends it with every subsequent request to prove their identity — without the server needing to look up a session in the database.

What does a JWT look like?

A JWT is three base64-encoded parts separated by dots:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9   ← Header (algorithm)
.eyJzdWIiOiJ1c2VyLTEyMyIsInVzZXJuYW1lIjoiam9obiIsInJvbGUiOiJVU0VSIn0
                                          ← Payload (your data)
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
                                          ← Signature (proves it's untampered)

The payload (middle part) can be decoded by anyone — it is just base64, not encrypted. The signature is the security: it proves the token was created by the server using a secret key and has not been modified.

JWT in this project

// back-end/util/jwt.ts

// What gets stored inside the token
export type JwtPayload = {
  sub: string;        // the user's ID ("subject")
  username: string;
  role: string;       // 'ADMIN' | 'REFEREE' | 'USER' | 'GUEST'
  teamId: string | null;
};

// Called when user logs in successfully → creates a new token
export function signAccessToken(payload: JwtPayload): string {
  const secret = process.env.JWT_SECRET; // secret key from .env
  return jwt.sign(payload, secret, { expiresIn: '7d' }); // valid for 7 days
}

// Called on every protected request → verifies the token is valid
export function verifyAccessToken(token: string): JwtPayload {
  try {
    return jwt.verify(token, secret) as JwtPayload;
  } catch {
    throw new UnauthorizedError('Your session token is invalid or expired.');
  }
}

The complete login flow

1. User fills in username + password on the login page
⬇
2. Front-end sends POST /api/auth/login with { username, password }
⬇
3. Back-end finds the user in DB, runs bcrypt.compare(password, hash)
⬇
4. If valid: jwt.sign(payload, secret) creates a token
⬇
5. Back-end returns { user, token } to the front-end
⬇
6. Front-end stores token in localStorage, updates React state
⬇
7. All future API requests include: Authorization: Bearer <token>

The authentication flow (subsequent requests)

// On every protected API request the front-end sends:
fetch('/api/matches/abc/status', {
  method: 'PATCH',
  headers: {
    'Authorization': `Bearer ${token}`, // ← the stored JWT
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({ status: 'IN_PROGRESS' }),
});

User Roles and Permissions

This project uses Role-Based Access Control (RBAC). Every user has one role, and each role has specific permissions.

RoleWho is this?Permissions
ADMIN System administrator Everything: manage users, assign referees, simulate rounds, reset tournament, update any match
REFEREE Match official Can only update their assigned matches: set status to IN_PROGRESS or FINISHED, add/edit goals
USER Registered fan Can view matches, standings, stats. Cannot modify anything.
GUEST Unregistered visitor Very limited read access. Cannot view match details or stats without logging in.

How roles are enforced in the API

// In the controller, routes are protected with requireRoles()
matchRouter.patch('/:matchId/status',
  authenticateToken,              // check: must have a valid JWT
  requireRoles('ADMIN', 'REFEREE'), // check: role must be ADMIN or REFEREE
  asyncHandler(async (req, res) => {
    // ... only reaches here if both checks pass
  })
);

// In the service, additional fine-grained checks exist
function assertMatchAccess(actor: RequestUser, refereeId: string | null) {
  if (actor.role === 'ADMIN') return; // admins bypass everything

  if (actor.role === 'REFEREE' && actor.id === refereeId) return; // own match

  throw new ForbiddenError('Only the assigned referee or admin can update this match.');
}

CORS — Cross-Origin Resource Sharing

Browsers block requests to a different domain/port by default (the same-origin policy). Since the front-end runs on port 8080 and the back-end on port 3001, the back-end must explicitly allow this.

// back-end/app.ts
app.use(cors({
  origin: process.env.CORS_ORIGIN?.split(',') ?? '*',
  // In .env: CORS_ORIGIN=http://localhost:8080
  credentials: true, // allow cookies / auth headers
}));

In development CORS_ORIGIN is set to http://localhost:8080. In production it should be your actual domain. Setting it to * (wildcard) means any website could call your API — avoid that in production.

Environment variables & secrets

Sensitive values like the JWT secret and database password are never hardcoded in the source code. They come from environment variables loaded from a .env file (which is excluded from git).

# back-end/.env (never commit this file!)
APP_PORT=3000
DATABASE_URL=postgresql://tournament:tournament@localhost:5432/tournament_manager
JWT_SECRET=your-super-secret-key-change-this
JWT_EXPIRES_IN=7d
CORS_ORIGIN=http://localhost:8080

The code reads them with:

import * as dotenv from 'dotenv';
dotenv.config(); // loads .env file

const secret = process.env.JWT_SECRET; // reads the value

Session management on the front-end

File: front-end/lib/session.ts and front-end/lib/useSession.tsx

The front-end stores the JWT in localStorage and manages it through a React Context called SessionContext.

// lib/useSession.tsx — the React session context
export function SessionProvider({ children }) {
  const [token, setToken] = useState(null);
  const [user, setUser] = useState(null);

  // On page load, restore from localStorage
  useEffect(() => {
    setToken(readStoredToken());
    setUser(readStoredUser());
  }, []);

  const value = {
    token,
    user,
    isAuthenticated: Boolean(token && user),
    setSession: (token, user) => { storeSession(token, user); setToken(token); setUser(user); },
    logout: () => { clearSession(); setToken(null); setUser(null); },
  };

  return <SessionContext.Provider value={value}>{children}</SessionContext.Provider>;
}

// In any component:
const { token, user, isAuthenticated, logout } = useSession();
