Chapter 08

Presentation Q&A Guide

Questions you are likely to be asked, with answers you can practice.

How to use this guide: Read each question, think of your own answer, then read the provided answer. Practice saying the answers out loud. You don't need to memorize them word-for-word — just understand the concept so you can explain it in your own words.

Project Overview Questions

❓ Can you explain what your project does in one sentence?

"Worldcup Manager 2026 is a web application where an administrator can manage a knockout football tournament — scheduling matches, assigning referees, tracking goals, and the system automatically advances the bracket when a round is completed."

❓ Who are the users of this application and what can they do?

There are four user roles:

  • Admin — can manage everything: users, matches, teams, simulate rounds, reset the tournament
  • Referee — can update only their assigned matches: set the status to in-progress or finished, record goals
  • User (registered fan) — can view matches, standings, and statistics
  • Guest (public) — very limited read access without logging in
❓ What is a "knockout tournament" format?

In a knockout tournament, the loser of each match is eliminated. This project has 4 rounds: Round of 16 (8 matches, 16 teams), Quarterfinal (4 matches, 8 teams), Semifinal (2 matches, 4 teams), and Final (1 match, 2 teams). After each round, the 8/4/2/1 winners are automatically placed into the next round's matches.

Technology Questions

❓ What technology stack did you use and why?

The back-end uses Node.js + Express for the API server, PostgreSQL as the database, and Prisma as the ORM. The front-end uses Next.js with React. Everything runs in Docker containers.

TypeScript is used throughout both front-end and back-end for type safety — it catches bugs at compile time before the code even runs.

❓ What is TypeScript and why would you use it over plain JavaScript?

TypeScript is a superset of JavaScript that adds static types. You declare what type each variable, parameter, and return value should be. The TypeScript compiler then checks that you never pass the wrong type — catching many bugs before the code even runs. It also makes code easier to understand because you can see exactly what shape a piece of data has just by looking at its type.

❓ What is Prisma and what problem does it solve?

Prisma is an ORM — Object-Relational Mapper. Without it, you would write raw SQL strings: "SELECT * FROM users WHERE id = $1". With Prisma, you write TypeScript: prisma.user.findUnique({ where: { id } }). This is type-safe (no typos in column names), easier to read, and automatically generates the SQL. Prisma also manages the database schema through a single schema.prisma file.

❓ What is React and why is it popular?

React is a JavaScript library for building user interfaces using components — reusable, self-contained pieces of UI. When data changes, React automatically updates only the parts of the page that need to change (this is called the virtual DOM). It is popular because it makes complex UIs easier to build and maintain, and there is a huge ecosystem of libraries around it.

❓ What is Next.js and how does it differ from plain React?

Next.js is a framework built on top of React. Plain React is just a UI library — you still need to add routing, server-side rendering, build tools, etc. yourself. Next.js provides all of that out of the box: file-based routing (every file in pages/ becomes a URL), server-side rendering, optimized builds, image optimization, and more.

❓ What is Docker and why did you use it?

Docker packages an application and all its dependencies into a container — a lightweight, isolated environment that runs the same on every machine. This project uses Docker Compose to start three containers at once: the PostgreSQL database, the Express back-end, and the Next.js front-end. Without Docker, every developer would need to manually install and configure Node.js, PostgreSQL, set environment variables, etc. With Docker, it is just one command: docker compose up.

Architecture Questions

❓ Can you explain the layered architecture of your back-end?

The back-end has four layers: (1) Controller — handles HTTP requests and routes, calls the service; (2) Service — contains the business logic and validation rules; (3) Repository/Prisma — executes database queries; (4) Domain Model — TypeScript classes with validation in their constructors. Each layer only talks to the one below it, which makes the code easier to test and change.

❓ What is a REST API?

REST (Representational State Transfer) is a convention for building web APIs. You design the API around resources (things like matches, players, users) and use standard HTTP methods to interact with them: GET to read, POST to create, PUT to replace, PATCH to partially update, DELETE to remove. The server and client communicate using JSON. In this project, the front-end calls the back-end's REST API to fetch and update data.

❓ Why did you use a layered architecture instead of putting everything in one file?

Separation of concerns — each layer has one clear job. This makes the code: (1) Testable — you can test business rules in the service without needing a real database; (2) Maintainable — if the database changes, only the Prisma queries need updating; (3) Understandable — when you look for a bug, you know which layer to look in based on what the bug is about.

Authentication Questions

❓ How does the login process work?

When a user submits their username and password, the front-end sends a POST request to /api/auth/login. The back-end looks up the user in the database, then uses bcrypt.compare() to check if the entered password matches the stored hash. If valid, it creates a JWT token (containing the user's ID, username, and role) signed with a secret key and returns it. The front-end stores this token in localStorage and sends it in the Authorization: Bearer header with every subsequent API request.

❓ Why are passwords not stored in plain text?

Because if the database is ever breached, attackers would have everyone's passwords. Instead, we use bcrypt hashing — a one-way function that transforms the password into a hash string. The hash cannot be reversed to get the original password. When someone logs in, we hash their entered password and compare the hashes. Bcrypt also adds a "salt" (random data) so the same password always produces a different hash, defeating rainbow table attacks.

❓ What is a JWT? How does it work?

JWT (JSON Web Token) is a compact, self-contained way to transmit information. It has three parts: a header (algorithm), a payload (the data — user ID, role, etc.), and a signature. The signature is created using a secret key — only the server with that key can create a valid signature. When the server receives a token, it verifies the signature. If it matches, the token data is trusted. This is stateless — the server doesn't need to store sessions in the database, it just verifies the token's signature on every request.

❓ What is the difference between 401 and 403 HTTP errors?

401 Unauthorized means "you are not logged in" — no valid token was provided. 403 Forbidden means "you are logged in, but you don't have permission to do this" — for example, a USER trying to access an ADMIN-only endpoint.

Business Logic Questions

❓ What are the match statuses and what do they mean?
  • PLANNED — match is scheduled but teams might not be confirmed yet
  • NOT_STARTED — teams are confirmed, countdown to kickoff
  • IN_PROGRESS — match is currently being played, goals can be recorded
  • FINISHED — final score has been entered by the referee
  • COMPLETED — the system has processed the result and advanced the bracket

A referee can only move status forward (e.g., NOT_STARTED → IN_PROGRESS), never backwards.

❓ How does automatic bracket advancement work?

When a referee marks a match as FINISHED and provides the score, the service layer checks if all matches in the current round are now FINISHED. If yes, it calls createNextRoundMatchesIfReady() which: (1) determines the winner of each match (higher score wins — draws are not allowed in knockout); (2) assigns those winners as the home/away teams for the next round's matches; (3) marks the current round matches as COMPLETED. This runs in a database transaction so it's atomic — either all assignments succeed or none do.

❓ How is the standings table calculated?

The calculateStandings function in tournamentService.ts loops over all FINISHED or COMPLETED matches and updates a standings row for each team: 3 points for a win, 1 for a draw, 0 for a loss. It also tracks goals for/against. Teams are then sorted by: points (descending) → goal difference → goals scored → alphabetical. This calculation happens entirely in JavaScript/TypeScript — no SQL is needed.

❓ Why can't a match end in a draw in this tournament?

Because it's a knockout tournament — every match must have a winner to advance the bracket. If a match ends in a draw, the roundProgressionService throws a ValidationError: "Knockout matches cannot end in a draw." This is a business rule enforced in the service layer.

❓ How are goals connected to the match score?

Goals are stored as individual records in the Goal table. When a goal is added or modified, the recalculateMatchScores() function counts all goals in the database for each team in that match and updates homeScore and awayScore on the Match record. This means the score is always derived from the stored goals — the score and goals are always consistent.

Testing Questions

❓ What tests did you write and why?

The tests in back-end/test/domain.test.ts test the domain model classes (Team, Player, Match). They verify that the validation rules work correctly — for example: creating a valid team succeeds, creating a team with an empty name throws the right error, a Match where home and away team are the same throws an error. These are called unit tests because they test a single unit (one class) in isolation, without needing a database or network.

❓ What is the difference between unit tests and integration tests?

Unit tests test one small piece of code in isolation (e.g., a class constructor). They are fast and don't need external systems. Integration tests test how multiple parts work together (e.g., does the API endpoint correctly store data in the database). Integration tests are slower because they need a real database. This project has unit tests for the domain models.

Code Explanation Questions

❓ What does async/await mean?

JavaScript is non-blocking — operations like database queries don't freeze the program, they run in the background. async marks a function as asynchronous. await pauses the execution of that async function until the awaited operation completes, then continues. This makes async code look and behave like synchronous code — much easier to read than callbacks or raw Promises.

// Without await (callback hell):
prisma.user.findUnique({ where: { id } }).then(user => {
  if (!user) throw new Error('not found');
  res.json(user);
}).catch(next);

// With async/await (clean and readable):
const user = await prisma.user.findUnique({ where: { id } });
if (!user) throw new Error('not found');
res.json(user);
❓ What is middleware in Express and how does it work?

Middleware is a function that sits between the incoming request and the route handler. It receives the request, can do something (check a token, parse a body, log the request), then either calls next() to pass control to the next middleware or handler, or calls next(error) to forward an error. In this project, authenticateToken and requireRoles are middleware functions that run before protected route handlers.

❓ What is a React Context and why is it used for session management?

Normally in React, if you want to share data between components, you pass it as props from parent to child. But if a deeply nested component needs the user's login status, you'd have to pass it through every intermediate component — called "prop drilling". React Context solves this: it creates a "global" store that any component in the tree can read from directly. The SessionContext wraps the entire app and provides the token, user, login, and logout functions to every component.

❓ What is SWR and why is it used for data fetching?

SWR is a React hook library for data fetching. The name stands for "stale-while-revalidate" — a caching strategy where it immediately shows cached (possibly old) data, and simultaneously re-fetches fresh data in the background. Compared to writing useEffect + fetch manually, SWR handles loading states, error states, automatic retry, deduplication (if 3 components need the same data, only 1 request is made), and cache invalidation via the mutate function.

❓ What is a Prisma transaction and when do you use it?

A transaction groups multiple database operations so they either all succeed or all fail together (this is called "atomicity"). In this project, when advancing the bracket, we update multiple match records. If one update fails halfway through, all the previous updates are rolled back — the database stays in a consistent state. Without a transaction, a partial update could leave some next-round matches with teams assigned and others still TBD.

❓ What are TypeScript enums and why use them?

An enum is a set of named constants. For example, MatchStatus is an enum with values PLANNED, NOT_STARTED, IN_PROGRESS, FINISHED, COMPLETED. Without an enum, you'd use plain strings — and a typo like "IN PROGRESS" (with a space) would cause a bug that's hard to find. With enums, TypeScript checks that you only use valid values. The enum is also defined in the Prisma schema, so the database enforces valid values too.

Critical Thinking / Improvement Questions

❓ What would you improve if you had more time?

Good honest answers you could give:

  • Add more comprehensive tests — integration tests for the API endpoints, not just unit tests for domain models
  • Add real-time updates using WebSockets — so the scoreboard updates live without refreshing
  • Add a proper group stage before the knockout rounds
  • Better input validation (e.g., password strength requirements)
  • Email notifications when a match is assigned to a referee
  • A more production-ready deployment (HTTPS, stronger secrets, cloud database)
❓ Is it a security issue that the JWT payload can be decoded by anyone?

No — and this is an important distinction. The JWT payload is base64-encoded, not encrypted — anyone can decode it. But the signature is what provides security. The signature is created with a secret key that only the server knows. If anyone modifies the payload (e.g., changes role from USER to ADMIN), the signature becomes invalid and the server rejects it. So the data in the token is readable but tamper-proof. Never put sensitive data like passwords in a JWT payload though.

❓ How would you explain the difference between the front-end role check and the back-end role check?

The front-end role check is a UX improvement — it hides restricted content from users who can't access it, providing a better experience. But it is not real security — a user could bypass it by using browser developer tools or making direct API calls. The back-end requireRoles middleware is the real security boundary — even if someone bypasses the front-end entirely and calls the API directly, the back-end will reject unauthorized requests with a 401 or 403 error.
