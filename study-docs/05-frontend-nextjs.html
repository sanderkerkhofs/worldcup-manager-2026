The _app.tsx wrapper

This special file wraps every page with shared providers and the Layout component:

// pages/_app.tsx
export default function App({ Component, pageProps }) {
  return (
    <SessionProvider>       {/* makes login state available everywhere */}
      <I18nProvider>         {/* makes translations available everywhere */}
        <Layout>             {/* the navbar + page wrapper */}
          <Component {...pageProps} />   {/* the actual page */}
        </Layout>
      </I18nProvider>
    </SessionProvider>
  );
}

React fundamentals used in this project

Components

A React component is a function that returns JSX (HTML-like syntax). Components can accept props (input data) and maintain state (internal data that can change).

// A simple component with props
function TeamBadge({ flag, name }: { flag: string; name: string }) {
  return (
    <span className="team-badge">
      {flag} {name}
    </span>
  );
}

// Using it in another component
function MatchCard({ match }) {
  return (
    <div>
      <TeamBadge flag={match.homeTeam.flag} name={match.homeTeam.name} />
      <span>{match.homeScore} — {match.awayScore}</span>
      <TeamBadge flag={match.awayTeam.flag} name={match.awayTeam.name} />
    </div>
  );
}

useState — local state

useState lets a component remember a value between re-renders and update the UI when it changes.

// From admin.tsx
const [message, setMessage] = useState<string | null>(null);
const [resettingMatches, setResettingMatches] = useState(false);

// When the reset button is clicked:
async function handleReset() {
  setResettingMatches(true);   // shows a loading indicator
  try {
    await resetMatches(token!);
    setMessage('Tournament reset successfully!');
    mutateOverview(); // re-fetches the overview data
  } catch (err) {
    setMessage(err.message);
  } finally {
    setResettingMatches(false); // hides loading indicator
  }
}

useEffect — side effects

useEffect runs code after the component renders. Typically used for: fetching data, reading localStorage, setting up timers.

// From useSession.tsx — load stored session on first render
useEffect(() => {
  setToken(readStoredToken()); // read from localStorage
  setUser(readStoredUser());
}, []); // empty [] = only run once when component mounts

useMemo — expensive calculations

useMemo caches a computed value and only recalculates when its dependencies change. Useful for sorting/filtering large lists.

// From index.tsx — sorted rounds, only recalculates when 'rounds' changes
const orderedRounds = useMemo(
  () => [...rounds].sort((a, b) => a.orderNumber - b.orderNumber),
  [rounds]
);

Data fetching with SWR

SWR is used throughout the front-end to fetch data from the API:

const { data: overview, error, isLoading, mutate } = useSWR(
  token ? ['overview', token] : null,  // ← cache key (null = don't fetch yet)
  () => getOverview(token)              // ← the actual fetch function
);

// 3 states to handle:
if (isLoading) return <p>Loading...</p>;
if (error)     return <p>Error: {error.message}</p>;
if (!overview) return null;

// Use the data:
return <p>{overview.competition.name}</p>;

// After a mutation (e.g., updating a match), re-fetch:
mutate(); // triggers SWR to re-call getOverview and update the UI

Why SWR instead of useEffect + fetch?

  • Deduplication — multiple components using the same key only cause one request
  • Caching — shows stale data immediately while re-fetching fresh data
  • Auto-retry — automatically retries on network errors
  • Less boilerplate — no manual loading/error state management

The API service layer

Files: front-end/services/ and front-end/lib/api.ts

Instead of calling fetch() directly in components, all API calls go through a clean service layer:

// lib/api.ts — a thin wrapper around fetch
const BASE_URL = process.env.NEXT_PUBLIC_API_URL ?? 'http://localhost:3001';

export const api = {
  get: async <T>(path: string, token?: string | null): Promise<T> => {
    const res = await fetch(BASE_URL + path, {
      headers: token ? { Authorization: `Bearer ${token}` } : {},
    });
    if (!res.ok) {
      const body = await res.json();
      throw new Error(body.error ?? 'Request failed');
    }
    return res.json();
  },

  post: async <T>(path: string, body: unknown, token?: string): Promise<T> => {
    const res = await fetch(BASE_URL + path, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
      },
      body: JSON.stringify(body),
    });
    // ...
  }
};
// services/competitionService.ts — uses the api wrapper
export function getOverview(token?: string | null) {
  return api.get<CompetitionOverview>('/api/competition', token);
}

export function updateMatchStatus(matchId: string, status: MatchStatus, token: string) {
  return api.patch<Match>(`/api/matches/${matchId}/status`, { status }, token);
}

Internationalisation (i18n)

File: front-end/lib/i18n.tsx

The app supports three languages: Dutch (NL), English (EN), and French (FR). The current language is stored in localStorage and applied via a React Context.

// All text in the app goes through the t() function
const { t, locale, setLocale } = useI18n();

// Instead of hardcoded text:
<h2>Huidige ronde</h2>

// You write:
<h2>{t('homeCurrentRound')}</h2>
//       ^ this key maps to "Huidige ronde" in NL,
//         "Current Round" in EN, "Tour Actuel" in FR
// The translations are defined in a big object:
const translations = {
  nl: {
    homeCurrentRound: 'Huidige ronde',
    navMatches: 'Wedstrijden',
    // ...
  },
  en: {
    homeCurrentRound: 'Current Round',
    navMatches: 'Matches',
    // ...
  },
  fr: {
    homeCurrentRound: 'Tour Actuel',
    navMatches: 'Matchs',
    // ...
  }
};

Role-based page access

Pages that require a specific role render a "restricted access" message if the user doesn't have permission. This is a pattern used throughout the app:

// pages/admin.tsx
export default function AdminPage() {
  const { token, user } = useSession();

  // Guard: if not logged in or not ADMIN, show restricted message
  if (!token || user?.role !== 'ADMIN') {
    const isLoggedIn = Boolean(token && user);
    return (
      <section>
        <h2>Restricted Access</h2>
        <p>{isLoggedIn ? 'You are not authorized.' : 'Please log in.'}</p>
        {!isLoggedIn && <Link href="/login">Go to Login</Link>}
      </section>
    );
  }

  // Only reaches here if user is ADMIN
  return <div>...admin content...</div>;
}

Note: This front-end check is a UX convenience, not a security measure. The real security is the requireRoles middleware on the back-end. Even if someone bypasses the front-end check, the API will still reject their requests.

The Layout component

File: front-end/components/Layout.tsx

The Layout wraps every page with a navigation bar and a consistent page container. It reads the session to show different nav links depending on the user's role.

  • Logged out: shows Login and Register
  • USER: shows Matches, Stats, Logout
  • REFEREE: shows the above + Referee area
  • ADMIN: shows the above + Admin area
