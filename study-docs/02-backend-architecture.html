Chapter 02

Backend Architecture

How the server is organised in layers, and what each layer does.

The Layered Architecture pattern

The back-end is organised into 4 layers. Each layer has one clear responsibility and only talks to the layer directly below it.

🌐 Controller Layer
← HTTP request comes in here
⬇ calls
⚙️ Service Layer
← business logic lives here
⬇ calls
🗃️ Repository / Prisma
← talks to the database
⬇ maps to
🏗️ Domain Model
← pure TypeScript classes with validation

Why separate layers? Because it makes code easier to test, change, and understand. If the database changes, only the repository layer needs to be updated. If a business rule changes, only the service layer changes.

Layer 1: Controller (Routes)

Files: back-end/controller/

The controller layer is the entry point for every HTTP request. Its only jobs are:

  1. Define which URL / HTTP method maps to which handler function
  2. Read data from the incoming request (req.body, req.params)
  3. Call the appropriate service function
  4. Send the response back to the client
  5. Protect routes using middleware (authentication / authorisation)

Controllers do not contain any business logic or database queries — that is the service layer's job.

// back-end/controller/matchController.ts

// GET /api/matches — anyone can see the match list
matchRouter.get('/', asyncHandler(async (_req, res) => {
  const matches = await listMatches();   // ← calls service
  res.json(matches);                     // ← sends JSON response
}));

// PATCH /api/matches/:matchId/status — only admins and referees
matchRouter.patch(
  '/:matchId/status',
  authenticateToken,                            // ← middleware: must be logged in
  requireRoles('ADMIN', 'REFEREE'),             // ← middleware: must have right role
  asyncHandler(async (req, res) => {
    const user = getAuthenticatedUser(req);
    const match = await updateMatchStatus(      // ← calls service
      req.params.matchId,                       // ← URL param
      req.body.status,                          // ← request body
      user
    );
    res.json(match);
  })
);

Layer 2: Service (Business Logic)

Files: back-end/service/

The service layer contains the real business logic of the application — the rules that make this a "World Cup manager" rather than just a generic data API. This is the most important layer.

Examples of business logic in the service layer:

  • A referee can only update their own assigned match
  • A match can only move from NOT_STARTED → IN_PROGRESS → FINISHED (not backwards)
  • A round is locked until all matches in the previous round are COMPLETED
  • When a round is finished, the winners are automatically placed into the next round's matches
  • Knockout matches cannot end in a draw
// back-end/service/matchService.ts

// Referees can only move a match forward through the status flow they control.
function assertStatusTransition(matchStatus, nextStatus, actor) {
  if (actor.role !== 'REFEREE') return; // Admins can do anything

  if (nextStatus !== 'IN_PROGRESS' && nextStatus !== 'FINISHED') {
    throw new ValidationError('Referees can only set a match to IN_PROGRESS or FINISHED.');
  }

  if (nextStatus === 'IN_PROGRESS'
      && matchStatus !== 'PLANNED'
      && matchStatus !== 'NOT_STARTED') {
    throw new ValidationError('Referees can only set IN_PROGRESS from PLANNED or NOT_STARTED.');
  }

  if (nextStatus === 'FINISHED' && matchStatus !== 'IN_PROGRESS') {
    throw new ValidationError('Referees can only set FINISHED from IN_PROGRESS.');
  }
}

Layer 3: Domain Model (Classes)

Files: back-end/model/

The domain model is a set of TypeScript classes that represent the core business entities. Each class's constructor validates its own data — if you try to create an invalid object, it throws a ValidationError immediately.

This is called "rich domain model" — the objects know and enforce their own rules.

// back-end/model/player.ts (simplified)
export class Player {
  public readonly id: string;
  public readonly shirtNumber: number;
  // ... other fields

  constructor({ id, shirtNumber, teamId, firstName, ...}: PlayerProps) {
    // Validation — constructor throws if rules are violated
    if (!teamId.trim()) {
      throw new ValidationError('Player must belong to a team.');
    }
    if (!Number.isInteger(shirtNumber) || shirtNumber <= 0) {
      throw new ValidationError('Shirt number must be a positive integer.');
    }

    // Assign the values (all fields are readonly — cannot be changed after creation)
    this.id = id;
    this.shirtNumber = shirtNumber;
    // ...
  }

  // Factory method: converts a Prisma database row into a Player object
  static from(prismaPlayer: PrismaPlayer): Player {
    return new Player({ ...prismaPlayer });
  }
}

Layer 4: Repository / Prisma

Files: back-end/repository/prisma/

The repository layer is the thin bridge between the service layer and the actual database. In this project, it is implemented directly using the Prisma client — there is no extra wrapper code needed because Prisma already provides a clean, type-safe API.

// back-end/repository/prisma/client.ts
import { PrismaClient } from '@prisma/client';

// A single shared Prisma client instance used across the whole app
export const prisma = new PrismaClient();

All database operations in the service layer look like:

// Find one record
const match = await prisma.match.findUnique({ where: { id: matchId } });

// Find many with filters
const matches = await prisma.match.findMany({
  where: { roundOrderNumber: 1 },
  orderBy: { matchDate: 'asc' },
});

// Create a record
await prisma.user.create({
  data: { username, passwordHash, role: 'USER' },
});

// Update a record
await prisma.match.update({
  where: { id: matchId },
  data: { status: 'IN_PROGRESS' },
});

// Run multiple operations atomically (all succeed or all fail)
await prisma.$transaction(async (tx) => {
  await tx.match.update({ ... });
  await tx.goal.deleteMany({ ... });
});

Middleware: the glue between layers

File: back-end/util/middleware.ts

Middleware functions run between the request arriving and the route handler. They can inspect / modify the request, then either pass it forward (next()) or reject it with an error.

authenticateToken

Reads the Authorization: Bearer <token> header, verifies the JWT, and attaches the decoded user to the request object.

export function authenticateToken(req, _res, next) {
  const header = req.headers.authorization;

  if (!header || !header.startsWith('Bearer ')) {
    return next(new UnauthorizedError('You must send a Bearer token.'));
  }

  const token = header.slice('Bearer '.length);
  const payload = verifyAccessToken(token); // throws if invalid/expired

  req.user = {
    id: payload.sub,
    username: payload.username,
    role: payload.role,
    teamId: payload.teamId,
  };

  next(); // passes control to the next middleware or route handler
}

requireRoles

A middleware factory — you call it with the allowed roles, and it returns a middleware function that checks if the logged-in user has one of those roles.

export function requireRoles(...roles: string[]) {
  return (req, _res, next) => {
    if (!req.user) return next(new UnauthorizedError());
    if (!roles.includes(req.user.role)) return next(new ForbiddenError());
    next();
  };
}

// Usage on a route:
matchRouter.patch('/:id/status',
  authenticateToken,              // step 1: must be logged in
  requireRoles('ADMIN', 'REFEREE'), // step 2: must have right role
  asyncHandler(handler)           // step 3: the actual logic
);

asyncHandler

Express doesn't automatically catch async errors. asyncHandler wraps an async function and forwards any thrown error to the error handler middleware.

export function asyncHandler(handler) {
  return (req, res, next) => {
    // If the async handler throws, .catch(next) sends the error to errorHandler
    Promise.resolve(handler(req, res, next)).catch(next);
  };
}

errorHandler

A global error handler. Every error thrown anywhere in the request pipeline ends up here. It returns a consistent JSON error response.

export function errorHandler(error, _req, res, _next) {
  if (error instanceof ApplicationError) {
    // Our custom errors have a specific status code (400, 401, 403, 404...)
    res.status(error.statusCode).json({ error: error.message });
    return;
  }
  // Unexpected errors → 500
  res.status(500).json({ error: 'An unexpected error occurred.' });
}

Complete request flow example

Let's trace what happens when a referee sends PATCH /api/matches/abc123/status with { "status": "IN_PROGRESS" }:

  1. Express router matches the URL and method → calls the matchRouter.patch handler chain
  2. authenticateToken middleware reads the Bearer token, verifies it, attaches req.user = { id, username, role: 'REFEREE', ... }
  3. requireRoles('ADMIN','REFEREE') checks req.user.role — passes because 'REFEREE' is in the list
  4. asyncHandler wraps the actual handler to catch async errors
  5. Controller handler reads req.params.matchId and req.body.status, calls updateMatchStatus(matchId, 'IN_PROGRESS', user)
  6. Service function updateMatchStatus: loads the match, checks the round is unlocked, validates the status transition for a referee, updates the DB
  7. Prisma runs the SQL UPDATE against PostgreSQL
  8. The updated match object is returned up through service → controller → res.json(match)
