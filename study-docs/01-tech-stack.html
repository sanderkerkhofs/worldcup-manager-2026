Technology Stack

Every tool used in this project, explained in plain language.

What is a "tech stack"?

A tech stack is the collection of technologies (programming languages, frameworks, libraries, databases, tools) that together make an application work. Think of it like the ingredients of a recipe — each one plays a specific role.

This project is a full-stack web application: it has a back-end (server + database) and a front-end (the website the user sees).

Language: TypeScript

What it is

A superset of JavaScript that adds types. Every valid JavaScript file is also valid TypeScript.

Why it matters

Catches bugs before you run the code. Your editor shows errors instantly. Autocomplete becomes far more powerful.

JavaScript is very flexible — you can pass anything anywhere. TypeScript adds type annotations so the compiler knows what shape every piece of data must have:

// Plain JavaScript — no type safety
function greet(name) {
  return "Hello " + name;
}

// TypeScript — name MUST be a string
function greet(name: string): string {
  return "Hello " + name;
}

// TypeScript catches this mistake at compile time, not at runtime:
greet(42); // ❌ Error: Argument of type 'number' is not assignable to 'string'

TypeScript is compiled to plain JavaScript before running. The tools ts-node (for development) and tsc (for production builds) do this compilation automatically.

Back-end Technologies

Node.js — the runtime

Node.js lets you run JavaScript/TypeScript outside the browser, on a server. It is single-threaded but non-blocking (async/await), meaning one Node.js process can serve thousands of requests without freezing.

Express.js — the web framework

Express is a minimal, unopinionated web framework for Node.js. It provides:

  • Routing — map a URL + HTTP method to a function
  • Middleware — intercept requests to add logic (e.g., auth check)
  • Response helpers — easily send JSON, set status codes
// back-end/app.ts — simplified
const app = express();

app.use(helmet());          // security headers
app.use(cors({ ... }));     // allow cross-origin requests
app.use(bodyParser.json()); // parse JSON request bodies

app.get('/status', (req, res) => {
  res.json({ message: 'Back-end is running...' });
});

app.use('/api/auth',        authRouter);       // login/register routes
app.use('/api/competition', competitionRouter); // tournament routes
app.use('/api/matches',     matchRouter);       // match routes
app.use('/api/players',     playerRouter);      // player routes

Database

PostgreSQL — the database

PostgreSQL is a relational database. Data lives in tables (like spreadsheets) and tables are linked together with foreign keys.

Example: a Goal row has a matchId column that references the Match table — so every goal "belongs to" a match.

Prisma — the ORM

An ORM (Object-Relational Mapper) lets you work with database records as regular TypeScript objects instead of writing raw SQL strings.

// Without Prisma — raw SQL (error-prone, no type checking)
const rows = await db.query("SELECT * FROM users WHERE id = $1", [userId]);

// With Prisma — type-safe TypeScript
const user = await prisma.user.findUnique({ where: { id: userId } });
//    ^ TypeScript knows the exact shape of 'user'

Prisma reads repository/prisma/schema.prisma — a single file that defines all tables (called models in Prisma). From that file it generates TypeScript types automatically.

Front-end Technologies

React — the UI library

React is a JavaScript library for building user interfaces using components. A component is a reusable piece of UI that has its own logic and can re-render automatically when data changes.

// A simple React component (TSX = TypeScript + JSX)
function ScoreCard({ homeTeam, awayTeam, homeScore, awayScore }: Props) {
  return (
    <div className="card">
      <span>{homeTeam}</span>
      <strong>{homeScore} — {awayScore}</strong>
      <span>{awayTeam}</span>
    </div>
  );
}

The JSX syntax (<div>...</div> inside JS) looks like HTML but it is actually JavaScript. The React compiler transforms it into function calls.

Next.js — the React framework

Next.js builds on top of React and adds essential features for a real application:

  • File-based routing — pages/login.tsx automatically becomes the /login URL
  • Server-side rendering (SSR) — pages can be rendered on the server for speed and SEO
  • Image optimization, fast refresh, and much more

SWR — data fetching

SWR is a React hook that fetches data from an API. It handles loading states, errors, and background re-fetching automatically.

// useSWR(cache-key, fetch-function)
const { data: overview, isLoading, error } = useSWR(
  ['overview', token ?? 'public'],
  () => getOverview(token)
);

if (isLoading) return <p>Loading...</p>;
if (error)     return <p>Error!</p>;

// 'overview' now has the data from the API
return <p>{overview.competition.name}</p>;

DevOps / Deployment

Docker

Docker packages your application (code + all dependencies + runtime) into a container — a lightweight, isolated environment that runs the same on every machine.

Without Docker: every developer must manually install Node.js, PostgreSQL, configure environment variables...

With Docker: one command — docker compose up — starts everything.

Docker Compose

Docker Compose orchestrates multiple containers as a group. Your docker-compose-fullstack.yml defines three services:

services:
  database:
    image: postgres:16-alpine       # Official PostgreSQL image
    environment:
      POSTGRES_DB: tournament_manager

  backend:
    build: ./back-end               # Build from the back-end Dockerfile
    depends_on:
      database:
        condition: service_healthy  # Only starts after DB is ready

  frontend:
    build: ./front-end              # Build from the front-end Dockerfile

Testing

Jest — the test framework

Jest is a JavaScript testing framework. You write test cases that describe expected behavior, run npm test, and Jest tells you which tests pass or fail.

// back-end/test/domain.test.ts
describe('Team domain validation', () => {
  it('creates a valid team', () => {
    const team = new Team({ name: 'Argentina', ... });
    expect(team.name).toBe('Argentina'); // ✅ passes if name equals 'Argentina'
  });

  it('rejects an empty team name', () => {
    expect(
      () => new Team({ name: '   ', ... })  // whitespace-only name
    ).toThrow('Team name is required.');     // ✅ passes if it throws this error
  });
});

The tests in this project verify the domain model validation rules — making sure a Team without a name, a Player with shirt number 0, etc., are all rejected with clear error messages.
